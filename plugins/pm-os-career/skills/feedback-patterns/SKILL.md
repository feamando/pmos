# Feedback Patterns

## When to Apply

- User gives feedback to a team member
- User drafts performance review comments
- User discusses coaching strategies
- User logs feedback entries in career tracking

## What to Do

1. **Be direct** — state what's working and what needs work. No sandwich method.
2. **Lead with specifics** — "In the OTP launch, you did X well because Y" not "Good job overall"
3. **Separate observation from interpretation** — describe the behavior, then explain the impact
4. **Make it actionable** — feedback without a path forward is just criticism
5. **Time it right** — capture feedback close to the event, not months later at review time

## Feedback Format

**Positive feedback:**
```
- **[Date]** | Positive | [Specific observation] | [Competency dimension] | [Source]
```

**Growth feedback:**
```
- **[Date]** | Growth | [Specific observation] — [What to do differently] | [Competency dimension] | [Source]
```

## Competency Mapping

When logging feedback, always map to a competency dimension. This builds evidence for career plans over time. Default dimensions are loaded from config; fall back to:

| Dimension | Indicators |
|-----------|------------|
| Speed & Agility | Fast delivery, bias for action, proactive problem-solving |
| Data-Drivenness | Metrics-driven decisions, evidence-based approach |
| Egolessness | Team-first, learning mindset, mentorship |
| Customer Centricity | Customer focus, value-driven decisions |
| Ownership | End-to-end responsibility, long-term thinking |

## Anti-Patterns

- "Good job" without specifics
- Feedback only during review cycles (should be continuous)
- Avoiding difficult conversations by being vague
- Comparing team members to each other instead of to the competency bar
