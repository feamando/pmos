# Reporting Patterns

## When to Apply
- Generating sprint reports, pupdates, quarterly updates, or any status document
- Writing executive summaries or leadership-facing content
- Structuring metric-heavy communications

## Structure Rules
1. **Lead with the conclusion** — headline metric or status first, details second
2. **Include WoW/YoY metrics** — always show week-over-week and year-over-year changes where data exists
3. **Explicit owners** — every action item and blocker has a named owner
4. **Status tags** — use (P0), (Critical), (In Progress), (Planned) for quick scanning
5. **Bold for emphasis** — key terms, names, statuses, and blockers are bold
6. **Tables for comparative data** — metrics, roadmaps, risk matrices use tables

## Metric Formatting
- Raw numbers with comma separators: `7,476 orders`
- Percentage changes with sign: `+172% YoY`, `-1% WoW`
- Dollar amounts with appropriate scale: `$167 CAC`, `$1.2M revenue`
- Always pair a number with its comparison period

## Report Types

### Sprint Report
- **Structure:** Team -> Delivered (clustered by priority) -> Planned -> Learnings -> Experiments
- **Priority levels:** Sprint Focus, Secondary Priority
- **Synthesis:** 1-3 sentences per cluster, business language, no ticket numbers in narrative
- **External format:** Numbered milestones with sub-feature bullets

### Performance Update (Pupdate)
- **Structure:** Headline -> Key Metrics Table -> Channel Breakdown -> Hypotheses -> Looking Ahead
- **Data density:** High — include raw numbers and % changes
- **Hypothesis-driven:** Always explain WHY a metric moved, not just the number

### Quarterly Update
- **Structure:** Executive Summary -> Systemic Blockers -> Learnings -> Roadmap -> Dependencies -> Hot Debates
- **Fill brackets:** Template sections in [brackets] require manual input
- **Root cause focus:** Blockers need root cause analysis, not just description
